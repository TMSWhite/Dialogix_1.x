Evaluating Triceps

It would be nice to ask questions that will let us easily address how Triceps fares in terms of existing theories and best practices. One of the drawbacks of the Schneiderman questionnaire is that it focuses on presentation issues that will ultimately be under the author's control, rather than Triceps'. For example, do we really care whether the fonts were easy to read and many of the questions under your Screen section. Also, we haven't authored the Help section, so we don't know how it will appear. Would it be silly to ask questions about issues we didn't plan to change?

It would also be helpful to make several questionnaires based upon the roles with which different users will interact with Triceps. For example:

  • Authors
    • Logical Flow
    • Sentence Content
    • Translation Ease
  • Validators
    • Ease of debugging
    • appropriateness / completeness of syntax error checking
    • ease of fixing the reported errors
    • ease of subdividing the validation task across multiple people
  • Management
    • ease of subdividing authoring, translating, and validation tasks across a team of people
    • ease of merging the different validated parts into the final product
    • ease of training
    • ease of deployment
    • ease of versioning
  • Users (Interviewers & Interviewees)
    • learning, speed, accuracy, retention, satisfaction
  • Analysts
    • ease of collating data
    • ease of analysis

Other generic questions on which it would be nice to elicit opinions include:

  • Ranking the importance of various features of interviews:
    • functionality
      • learning, retention, accuracy, speed, satisfaction
    • presentation
      • how would a flashier design aide or hinder the interview process?
      • what input styles are best for different datatypes
        • style - radio, radio2, combo, list
        • size - should list boxes be limited to a certain number of entries, or should all be shown?
    • expectations
      • did the system ever surprise you - did something unexpected (gulf of execution, evalution, astonishment)
  • General comments
    • what did you most like about the system?
    • what did you most dislike?
    • how would you like to see the system enhanced?

Below is a summary of notes I culled from Schneiderman, Justin's course, and Stephen's Intro course. We might consider asking questions that let us evaluate Triceps in some of the terms listed below:

Theories - did we utilize and/or meet them?

  • Usability Theories
    • Principle I - Recognize the Diversity
      • User Profiling
        • Novice / first-time
          • make successful, informative feedback or tutorials
        • Knowledgeable-intermittent
          • oderly/consistent structure/terminology
          • high interface apparancy
          • protection from danger
        • Expert/Frequent
          • non-disturbing feedback
          • macros
      • Level-Structured
        • allow users to gradually move to more expert interface
      • Task Profiling
        • make most common tasks fastest/easiest to access/do
      • Interaction Style
        • direct manipulation
        • menu selection
        • form fill-in
        • command language
        • natural language
    • Principle II - Eight Golden Rules
      • consistency
      • shortcuts for frequent users
      • informative feedback
      • closure
      • error prevention / handling
      • easy reversal
      • internal locus of control
      • reduce short term memory load
    • Principle III - Prevent Errors
      • correct matching pairs (e.g. auto-closure of parentheses, detection of syntax errors, detection of HTML errors)
      • complete sequences - macros, alerting if a sequence (e.g. a dialog) is incomplete
      • correct commands - good feedback for command languages, not letting any commands break the system
    • Data Display Guidelines
      • consistency
      • efficient information assimilation
      • compatability of display with entry
      • flexibility - mouse & keyboard
      • minimal memory load
      • present data only if it will assist (e.g. no gnats)
      • getting attention - do the error messages make it clear what went wrong and what needs to be done
  • Other Goals
    • minimize gulf of execution
    • minimize gulf of evaluation
    • obey law of least astonishment
    • minimize distractions
    • minimize anxiety
    • provide informative feedback
    • avoid errors
    • avoid slow response times
    • pass monkey test (no actions should ever crash the system)
  • Information Search
    • Overview
    • Zoom
    • Filter
    • Details on Demand
    • Relate
    • History
    • Extract

Testing - how do we rate?

  • Evaluation during active use
    • focus groups, interviews
    • continuous user performance logging
      • user planning time
      • response time
      • user think time
    • online suggestion boxes, email, newsgroups
  • Ethnographic
    • critical incidents / memorable problems
    • structured observations
    • time & motion studies
    • scenario walkthroughs
  • Form / Data entry Guidelines
    • What & Why - give context of what to expect
    • Labels - in preferred lexicon / language
    • Grouping
    • Navigation
    • Error handling - monkey test
    • Error prevention
      • mask to limit type entry
      • hint/prototype (e.g. SSN: ###-##-####)
    • Flag mandatory vs. optional
